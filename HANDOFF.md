# Fantasy Hockey Score Tracker - Development Handoff

## Project Overview
Building a system to track how Yahoo Fantasy Hockey matchup scores (current and **live projected**) change throughout each week, then visualize momentum shifts with ESPN-style graphs similar to what the ESPN app provides for live sports.

**Why this matters:** Yahoo Fantasy shows current scores and projected final scores, but these projections change in real-time as games progress. Currently, there's no way to see *how* those projections changed over the course of a week. This tool captures that data and creates visualizations showing momentum shifts (e.g., "I was projected to lose at 2pm, but after my players had a hot night, I'm now projected to win").

## CRITICAL TECHNICAL DISCOVERY 🚨

### The Live Projection Problem & Solution

**The Issue:**
Yahoo's API returns TWO different projection fields that serve different purposes:

1. **`team_projected_points.total`**
   - Static projection calculated at the beginning of the week
   - Does NOT update during games
   - Essentially useless for tracking momentum

2. **`team_live_projected_points['total']`**
   - Real-time projection that updates as games progress
   - Shows actual momentum shifts
   - **This is what we need!**

**Example from actual data:**
- Static projection: 210.85 (says user will lose)
- Live projection: 243.82 (says user will win!)
- The 30+ point difference shows why this matters!

**The Solution:**
The live projection is NOT exposed as a direct property by yfpy. It exists in the internal `_extracted_data` dictionary:

```python
# DON'T USE (static):
team1_projected = float(team1.team_projected_points.total)

# DO USE (live, real-time):
if hasattr(team1, '_extracted_data'):
    live_proj = team1._extracted_data.get('team_live_projected_points')
    if live_proj and 'total' in live_proj:
        team1_projected = float(live_proj['total'])
```

**Why this works:**
- `_extracted_data` contains the raw API response before yfpy processes it
- Yahoo includes `team_live_projected_points` in the raw data
- yfpy doesn't expose it as a clean property, so we access the internal dict

**Implementation location:** `collect_matchup_data.py` lines ~59-72

## API Rate Limits & Best Practices

### Yahoo Fantasy API Limits
- **Undocumented officially** but community experience suggests:
  - ~20-50 requests per hour is safe
  - Excessive rapid requests (100+ in short time) trigger temporary blocks
  - Blocks last ~15-30 minutes then auto-clear

### Our Usage Pattern
- **Current:** Manual execution = ~6-12 API calls per day
- **Planned:** Every 2-4 hours = ~6-12 calls per day (well within limits)
- **Peak usage:** Every hour during games = ~24 calls per day (still safe)

### Planned Collection Schedule
**Intelligent timing based on NHL game patterns:**

**Weekdays (Monday-Friday):**
- 12:00 PM - 7:00 PM: Every 4 hours (off-peak)
- 7:00 PM - 11:30 PM: Every 1 hour (games happening)
- 11:30 PM - 12:00 AM: Every 2 hours (late games finishing)
- 12:00 AM - 12:00 PM: Every 6 hours or skip (overnight)

**Weekends (Saturday-Sunday):**
- 12:00 PM - 1:00 PM: Every 2 hours (pre-games)
- 1:00 PM - 11:30 PM: Every 1 hour (games happening)
- 11:30 PM - 12:00 PM: Every 6 hours (overnight)

**NHL Game Times (EST):**
- Weekday starts: 7:00 PM
- Weekend starts: 1:00 PM
- Latest starts: 10:30 PM (West Coast)
- Games end: ~2.5-3 hours after start

**Implementation Note:** Start with hardcoded schedule for MVP. Later enhancement could check NHL schedule API for actual game days.
